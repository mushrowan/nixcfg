# nixcfg schema

nixcfg consumes standard [JSON Schema (draft 2020-12)](https://json-schema.org/draft/2020-12)
with a small set of extension properties prefixed `x-nixcfg-`.

## extensions

| extension | on | effect |
|---|---|---|
| `x-nixcfg-name` | root | service/program name (used for module path) |
| `x-nixcfg-secret` | property | marks field as a secret |
| `x-nixcfg-port` | property | overrides integer type to `types.port` |
| `x-nixcfg-path` | property | overrides string type to `types.path` |
| `x-nixcfg-skip` | property | omits field from nix module options |
| `x-nixcfg-description` | property | overrides description text |
| `x-nixcfg-example` | property | overrides single example value |
| `x-nixcfg-config-format` | root | `toml` / `json` / `yaml` for `mkConfigFile` |

### secrets (`x-nixcfg-secret`)

when a property has `"x-nixcfg-secret": true`:

- **name** gets a `_path` suffix (`discord_token` -> `discordTokenPath` in camelCase)
- **type** becomes `path` (or `nullOr path` if nullable)
- **description** is prefixed with "path to file containing "
- **defaults** are dropped (optional secrets default to `null`)

the idea: your app reads the secret from a file at runtime, so nix
only needs to know the file path, never the secret value

### ports (`x-nixcfg-port`)

when a property has `"x-nixcfg-port": true`, the nix type becomes
`types.port` (unsigned 16-bit integer) regardless of the JSON Schema
integer format

### paths (`x-nixcfg-path`)

when a property has `"x-nixcfg-path": true`, the nix type becomes
`types.path`. composes with nullable (`["string", "null"]` ->
`nullOr path`). schemars emits `"format": "path"` for `PathBuf`
fields; that form is auto-detected as path without needing the
explicit extension

### skipped fields (`x-nixcfg-skip`)

when a property has `"x-nixcfg-skip": true`, it is omitted from
`optionsFromSchema` and `mkModule` output. the property still appears
in the schema itself, so other consumers (cli arg generation, config
file writing, docs) can see it

useful for runtime-only fields that shouldn't appear in a NixOS
module (callback handles, shared state, injected runtime context)
but need to be part of the rust config struct for the program

### config file format (`x-nixcfg-config-format`)

used by `mkConfigFile` to pick a serialiser. valid values: `toml`,
`json`, `yaml`. `mkConfigFile` throws if this key is missing

### description and example overrides

rust doc comments tend to be terse and code-facing ("cwd for hook
commands"). nix option descriptions want to be prose and
example-heavy ("working directory used when invoking configured
lifecycle hooks. defaults to the directory the app was launched
from"). trying to write to both audiences in a single `///` comment
doesn't scale for a large config

- `"x-nixcfg-description": "..."` overrides the `description` field
  when building the nix option. the original `description` stays in
  the schema for other consumers
- `"x-nixcfg-example": value` overrides the first of `examples`
  when building the nix option's `example` attribute

both are opt-in per field. if neither is set, behaviour is
unchanged: `description` passes through, `examples[0]` becomes
`example`

### validation constraints

nixcfg maps standard JSON Schema validation keywords to nix type
checks so invalid values are rejected at `evalModules` time:

- **strings**:
  - `pattern: "..."` → `types.strMatching "..."`
  - `minLength: n` / `maxLength: n` → composed via `addCheck` with
    `builtins.stringLength`
  - all three compose: a string with pattern + minLength + maxLength
    gets a `strMatching` base with two length checks on top
- **integers**:
  - `format: "uint8"` / `"uint16"` / `"uint32"` / `"int8"` /
    `"int16"` / `"int32"` → bounded `types.ints.u8` etc (see type
    mapping table below)
  - `minimum` + `maximum` (no format) → `types.ints.between lo hi`
  - `minimum: 0` alone → `types.ints.unsigned`

schemars emits these keywords automatically from
`#[schemars(regex(pattern = "..."))]`, `#[schemars(length(min =, max
=))]`, and the rust primitive type itself (`u16` → `"format":
"uint16"` + `minimum`/`maximum`)

## type mapping

| JSON Schema | nix type |
|---|---|
| `"type": "string"` | `types.str` |
| `"type": "string", "enum": [...]` | `types.enum [...]` |
| `"type": "string", "pattern": "..."` | `types.strMatching "..."` |
| `"type": "string", "minLength": n` | `types.str` + length check (composes with pattern) |
| `"type": "string", "maxLength": n` | `types.str` + length check (composes with pattern) |
| `"type": "string", "format": "path"` | `types.path` (schemars PathBuf) |
| `"type": "boolean"` | `types.bool` |
| `"type": "integer"` | `types.int` |
| `"type": "integer", "format": "uint8"` | `types.ints.u8` |
| `"type": "integer", "format": "uint16"` | `types.ints.u16` |
| `"type": "integer", "format": "uint32"` | `types.ints.u32` |
| `"type": "integer", "format": "int8"` | `types.ints.s8` |
| `"type": "integer", "format": "int16"` | `types.ints.s16` |
| `"type": "integer", "format": "int32"` | `types.ints.s32` |
| `"type": "integer", "x-nixcfg-port": true` | `types.port` |
| `"type": "integer", "minimum": lo, "maximum": hi` | `types.ints.between lo hi` |
| `"type": "integer", "minimum": 0` | `types.ints.unsigned` |
| `"type": "array", "items": T` | `types.listOf T` |
| `"type": "number"` | `types.float` (f32/f64) |
| `"type": "object", "properties": {...}` | `types.submodule {...}` |
| `"type": "object", "additionalProperties": T` | `types.attrsOf T` |
| `"type": ["T", "null"]` | `types.nullOr T` |
| `"anyOf": [A, B]` (non-null variants) | `types.either A B` (chained for more) |
| `"anyOf": [T, {"type": "null"}]` | `types.nullOr T` |
| `"$ref": "#/$defs/Foo"` | resolved and mapped recursively |

format strings match rust primitive names, so schemars fields like
`u16` / `i32` / `usize` get the right bounded nix type
automatically. for types without a bounded nix equivalent (`u64`,
`i64`, `usize`), the fallback order is format → bounds → unsigned →
int

## naming

- schema properties: **snake_case** (language-neutral convention)
- nix options: configurable via `naming` (default **camelCase**)
- CLI flags: configurable via `output` (default **kebab-case**)
- env vars: configurable via `output` (default **SCREAMING_SNAKE_CASE**)
- config file keys: configurable via `output` (default **snake_case**)

## example

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Config",
  "description": "my service configuration",
  "type": "object",
  "x-nixcfg-name": "my-service",
  "properties": {
    "data_dir": {
      "type": "string",
      "description": "data directory",
      "default": "/var/lib/my-service"
    },
    "listen_port": {
      "type": "integer",
      "format": "uint16",
      "minimum": 0,
      "maximum": 65535,
      "description": "port to listen on",
      "default": 8080,
      "x-nixcfg-port": true
    },
    "api_token": {
      "type": "string",
      "description": "API authentication token",
      "x-nixcfg-secret": true
    },
    "log_level": {
      "$ref": "#/$defs/LogLevel",
      "description": "log level",
      "default": "info"
    }
  },
  "required": ["data_dir", "listen_port", "api_token", "log_level"],
  "$defs": {
    "LogLevel": {
      "type": "string",
      "enum": ["trace", "debug", "info", "warn", "error"]
    }
  }
}
```

this generates nix options:

```nix
{
  dataDir = mkOption { type = types.str; default = "/var/lib/my-service"; };
  listenPort = mkOption { type = types.port; default = 8080; };
  apiTokenPath = mkOption { type = types.path; description = "path to file containing ..."; };
  logLevel = mkOption { type = types.enum [ "trace" "debug" "info" "warn" "error" ]; default = "info"; };
}
```

## rust driver

with schemars, annotate your structs directly:

```rust
use schemars::JsonSchema;
use serde::Serialize;

#[derive(JsonSchema, Serialize)]
/// my service configuration
struct Config {
    /// data directory
    #[serde(default = "default_data_dir")]
    data_dir: String,

    /// port to listen on
    #[schemars(extend("x-nixcfg-port" = true))]
    listen_port: u16,

    /// API authentication token
    #[schemars(extend("x-nixcfg-secret" = true))]
    api_token: String,

    /// log level
    log_level: LogLevel,
}

#[derive(JsonSchema, Serialize)]
enum LogLevel { Trace, Debug, Info, Warn, Error }
```

emit the schema:

```rust
use nixcfg::NixSchema;

let schema = NixSchema::from::<Config>("my-service");
println!("{}", schema.to_json_pretty());
```

### `#[nixcfg]` attribute macro

for configs with lots of annotated fields, the raw `#[schemars(extend(...))]`
form gets verbose. the `nixcfg` crate re-exports a `#[nixcfg]` attribute
macro that rewrites concise `#[nixcfg(...)]` field attributes into the
appropriate `#[schemars(extend(...))]` form

```rust
use nixcfg::{nixcfg, JsonSchema};
use serde::Serialize;

#[nixcfg]
#[derive(JsonSchema, Serialize)]
struct Config {
    #[nixcfg(secret)]
    api_key: String,

    #[nixcfg(port)]
    listen_port: u16,

    #[nixcfg(path)]
    data_dir: std::path::PathBuf,

    #[nixcfg(skip)]
    runtime_handle: std::sync::Arc<()>,

    // combinable
    #[nixcfg(secret, path)]
    pem_path: String,

    // key=value for description / example overrides
    #[nixcfg(
        description = "long prose explanation aimed at nix users",
        example = "/var/lib/service"
    )]
    hooks_cwd: String,
}
```

supported flags: `secret`, `port`, `path`, `skip`

supported key=value pairs: `description`, `example`

the macro must run before the `JsonSchema` derive, so `#[nixcfg]` goes
above `#[derive(...)]`

### foreign types via `schema_with`

for types you can't (or don't want to) derive `JsonSchema` on locally
-- e.g. types from a third-party crate -- use schemars's
`#[schemars(schema_with = "fn_path")]` to hand-roll the schema
fragment. nixcfg extensions are just JSON keys, so they pass through
untouched:

```rust
use nixcfg::{JsonSchema, schemars};
use serde::Serialize;

// a type from some other crate we can't modify
#[derive(Serialize)]
struct RemoteHandle { /* ... */ }

fn remote_handle_schema(_g: &mut schemars::SchemaGenerator) -> schemars::Schema {
    serde_json::from_value(serde_json::json!({
        "type": "object",
        "properties": {
            "endpoint": { "type": "string" }
        },
        "x-nixcfg-skip": true,
        "description": "runtime connection handle, populated at boot"
    })).unwrap()
}

#[derive(JsonSchema, Serialize)]
struct Config {
    // no doc comment: schemars would apply it as the field description,
    // overriding whatever schema_with produces. leaving it off lets the
    // hand-rolled description win
    #[schemars(schema_with = "remote_handle_schema")]
    handle: RemoteHandle,
}
```

any `x-nixcfg-*` keys in the returned fragment are honoured as
normal

## gleam driver

the `nixcfg` gleam package provides a builder DSL. gleam doesn't
have macros so schemas are constructed explicitly, one property at
a time

```gleam
import gleam/io
import gleam/json
import nixcfg

pub fn main() {
  nixcfg.new("myapp", "my service configuration")
  |> nixcfg.prop(
    "data_dir",
    nixcfg.string()
      |> nixcfg.default(json.string("/var/lib/myapp"))
      |> nixcfg.description("data directory"),
  )
  |> nixcfg.prop("listen_port", nixcfg.u16() |> nixcfg.port())
  |> nixcfg.prop(
    "api_token",
    nixcfg.string()
      |> nixcfg.secret()
      |> nixcfg.description("API authentication token"),
  )
  |> nixcfg.to_json
  |> io.println
}
```

constructors match the type mapping table: `string`, `boolean`,
`integer`, `unsigned`, `u8` / `u16` / `u32`, `i8` / `i16` / `i32`,
`enum_of`, `list_of`, `nullable`. for anything the builder can't
express, `nixcfg.raw(json.object([...]))` is an escape hatch that
passes the JSON fragment through to nixcfg unchanged

modifiers: `default`, `description`, `example`, `pattern`,
`min_length`, `max_length`, `secret`, `port`, `path`, `skip`,
`override_description`, `override_example`

nullable scalar properties are automatically excluded from the
schema's `required` list (nix treats them as optional). skipped
properties (`nixcfg.skip`) are also excluded from required
